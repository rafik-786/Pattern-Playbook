Real-World Analogies

Your Phone's Settings App ‚Äî Deep Mapping

Think about the Settings app on your phone. There's only ONE Settings app ‚Äî you can't install a second copy. No matter where you open it from (home screen, notification bar, or another app), you always get the SAME Settings app with the SAME data. If you change the Wi-Fi password in Settings, every app on your phone sees that new password immediately. You don't need to open separate Settings for each app ‚Äî there's one shared instance that everyone uses.

  • You never "create" a new Settings app ‚Äî your code never calls new on a Singleton
  • Everyone on the phone shares the same Settings ‚Äî every service in your app shares the same instance
  • Changes in one place are visible everywhere ‚Äî that's the whole point of having one shared object

Six Real-World Use Cases
1. Database Connection Pool

Your app shares a fixed set of database connections instead of opening a new one for every request. One Singleton manages the pool ‚Äî lending out connections and taking them back.

2. App Configuration

Your app reads the config file once at startup and keeps it in memory. Every part of the app gets settings from the same place ‚Äî no repeated file reads.

3. Logging

All parts of your app write logs through one shared logger. This ensures logs go to the same file in the right order, not scattered across 50 different writers.

4. Cache

Your app stores frequently used data in one shared cache. If every service had its own cache, the same data would be loaded from the database multiple times ‚Äî wasteful.

5. Feature Flags

Your app checks one central place to decide if a feature is ON or OFF. If each request checked independently, you'd get inconsistent behavior ‚Äî some users see the feature, some don't.

6. Rate Limiter

Your app uses one counter to track how many API calls a user has made. If each server thread had its own counter, users could bypass the limit by hitting different threads.

GoF Formal Definition

"Ensure a class has only one instance, and provide a global point of access to it."

The pattern solves two distinct problems: (1) it guarantees a class has just a single instance ‚Äî useful when exactly one object is needed to coordinate actions across the system; and (2) it provides a global access point to that instance ‚Äî avoiding the need to thread the instance through every call stack manually.

Singleton ‚Äî All 4 Canonical Implementations

NaiveSingleton.cs ‚Äî NOT THREAD-SAFE
// ‚ùå BROKEN: Race condition when two threads call Instance simultaneously
public sealed class NaiveSingleton
{
    private static NaiveSingleton? _instance;

    private NaiveSingleton() { }

    public static NaiveSingleton Instance
    {
        get
        {
            // Thread A checks: _instance == null ‚Üí YES
            // Thread B checks: _instance == null ‚Üí YES (before A assigns!)
            // Result: TWO instances created ‚Äî singleton contract broken
            if (_instance == null)
                _instance = new NaiveSingleton();
            return _instance;
        }
    }

    public string DoWork(string input) => $"Processed: {input}";
}

// Why this breaks:
// 1. No memory barrier ‚Äî Thread B may not see Thread A's write
// 2. No mutual exclusion ‚Äî both threads enter the if-block
// 3. No atomicity ‚Äî check + assign is two operations, not one
DclSingleton.cs ‚Äî Thread-Safe (Classic)
// ‚úÖ Thread-safe via double-checked locking
// Use when you need explicit control over initialization logic
public sealed class DclSingleton
{
    private static volatile DclSingleton? _instance;
    private static readonly object _lock = new();

    private DclSingleton()
    {
        // Expensive initialization goes here
        // This runs EXACTLY once, even under contention
    }

    public static DclSingleton Instance
    {
        get
        {
            // First check ‚Äî fast path (no lock needed after init)
            if (_instance != null) return _instance;

            lock (_lock)
            {
                // Second check ‚Äî inside lock prevents duplicate creation
                // Thread B enters lock AFTER Thread A constructed ‚Üí sees non-null
                _instance ??= new DclSingleton();
            }
            return _instance;
        }
    }

    public string DoWork(string input) => $"Processed: {input}";
}

// volatile keyword ensures:
// 1. Writes are visible to all threads immediately (memory barrier)
// 2. Prevents instruction reordering around the field
// Without volatile: Thread B could read a partially-constructed object
LazySingleton.cs ‚Äî Thread-Safe (Recommended for Libraries)
// ‚úÖ Thread-safe via Lazy<T> ‚Äî preferred for self-managed singletons
// Framework handles ALL locking internally
public sealed class LazySingleton
{
    // LazyThreadSafetyMode.ExecutionAndPublication:
    //   - Only ONE thread executes the factory
    //   - All other threads block until the value is ready
    //   - If factory throws, next call retries (exception is NOT cached)
    private static readonly Lazy<LazySingleton> _lazy =
        new(() => new LazySingleton(),
            LazyThreadSafetyMode.ExecutionAndPublication);

    private LazySingleton()
    {
        // Expensive initialization goes here
        // Guaranteed to run exactly once
    }

    public static LazySingleton Instance => _lazy.Value;

    // Check if instance has been created without triggering creation
    public static bool IsCreated => _lazy.IsValueCreated;

    public string DoWork(string input) => $"Processed: {input}";
}

// Why Lazy<T> is better than manual DCL:
// 1. No volatile field ‚Äî Lazy handles memory barriers internally
// 2. No lock object ‚Äî Lazy manages its own synchronization
// 3. No double-check pattern ‚Äî Lazy encapsulates the logic
// 4. Exception handling ‚Äî configurable retry on factory failure
DiSingleton.cs ‚Äî DI Container (Modern Best Practice)
// ‚úÖ Modern approach: DI container manages the singleton lifetime
// The class itself has NO static state ‚Äî it's just a normal class
// Simple record to expose internal state (read-only snapshot)
public record ServiceState(IReadOnlyDictionary<string, int> Counts);

public interface ISingletonService
{
    string DoWork(string input);
    ServiceState GetState();
}

public sealed class SingletonService : ISingletonService
{
    private readonly ILogger<SingletonService> _logger;
    private readonly ConcurrentDictionary<string, int> _state = new();

    // Normal constructor injection ‚Äî NOT private
    // DI container ensures only ONE instance is ever created
    public SingletonService(ILogger<SingletonService> logger)
    {
        _logger = logger;
        _logger.LogInformation("SingletonService created (this runs once)");
    }

    public string DoWork(string input)
    {
        _state.AddOrUpdate(input, 1, (_, count) => count + 1);
        return $"Processed: {input} (call #{_state[input]})";
    }

    public ServiceState GetState() =>
        new(_state.ToDictionary(kv => kv.Key, kv => kv.Value));
}

Why volatile is Critical
Without volatile, the CPU or JIT compiler may reorder instructions. Thread B could observe a non-null _instance pointer before the constructor body has finished executing. volatile inserts a memory barrier preventing this read reordering ‚Äî ensuring Thread B sees a fully-initialized object.

Junior vs Senior Thinking

Scenario 1 ‚Äî Configuration Manager

Both junior and senior developers need to load application configuration. The way they approach this reveals fundamentally different mental models about resource management (how carefully code uses memory, files, and connections), testability (how easy it is to write automated tests), and separation of concerns (keeping each class responsible for only one job).

Junior Approach
public class ConfigService
{
    public Dictionary<string, string> Settings { get; }

    public ConfigService()
    {
        // File I/O on every instantiation!
        var json = File.ReadAllText("appsettings.json");
        Settings = JsonSerializer
            .Deserialize<Dictionary<string, string>>(json)!;
    }

    public string Get(string key) => Settings[key];
}

public class OrderService
{
    public void ProcessOrder()
    {
        var config = new ConfigService(); // new instance every call!
        var connectionString =
            config.Get("Database:ConnectionString");
    }
}

public class InventoryService
{
    public void CheckStock()
    {
        var config = new ConfigService(); // another file read!
        var apiUrl = config.Get("Inventory:ApiUrl");
    }
}
VS
Senior Approach
public interface IConfigService
{
    string Get(string key);
    T GetSection<T>(string sectionName) where T : class, new();
}

public class ConfigService : IConfigService
{
    private readonly IConfiguration _configuration;
    public ConfigService(IConfiguration configuration)
        => _configuration = configuration;
    public string Get(string key)
        => _configuration[key]
            ?? throw new KeyNotFoundException(
                $"Config key '{key}' not found.");
    public T GetSection<T>(string sectionName) where T : class, new()
    {
        var section = new T();
        _configuration.GetSection(sectionName).Bind(section);
        return section;
    }
}

// DI Registration (once at startup)
builder.Services.AddSingleton<IConfigService, ConfigService>();

// Usage: injected, not constructed
public class OrderService
{
    private readonly IConfigService _config;
    public OrderService(IConfigService config) => _config = config;
    public void ProcessOrder()
    {
        var connectionString = _config.Get("Database:ConnectionString");
    }
}

Why Junior Is Wrong ‚Äî 5 Problems

Performance Waste
Every time any part of the app needs a config value (like a database URL), the junior's code opens the file, reads the entire JSON, parses it, builds a dictionary ‚Äî then throws it all away. Imagine a busy app handling 1,000 requests per second: that's 1,000 unnecessary disk reads per second for data that hasn't changed. It's like re-reading an entire book every time you need to recall one sentence.
Read Once, Reuse Forever
The senior's code reads the config file exactly once when the app starts. That parsed data lives in memory for the entire app lifetime. Every subsequent call just looks up a value from memory ‚Äî which takes nanoseconds instead of milliseconds. It's like memorizing the answer on day one and never opening the book again.
Inconsistent State
Picture this: your app is running, and someone updates the config file during a deployment. Request A reads the old file, and Request B (arriving 1ms later) reads the new file. Now two requests in the same moment are using different database URLs or feature flags. This split-brain (where two parts of the same app disagree on which data is current) situation causes bugs that are nearly impossible to reproduce or debug.
One Truth, Zero Conflicts
Since the config is loaded once into a single shared instance, every request sees the exact same data. No race conditions (two threads try to read/write the same data at the same time, producing unpredictable results), no split-brain. If you need to pick up config changes at runtime, the senior uses IOptionsMonitor<T> (a built-in ASP.NET feature that watches the config file for changes) which handles reloads atomically (happens as one uninterruptible operation ‚Äî all of it completes, or none of it does) ‚Äî all requests switch to the new config at the same time.
Memory Bloat
Each new ConfigService() creates a brand-new Dictionary object on the heap (the region of memory where objects live while your app runs). Under load, thousands of these short-lived objects pile up and the Garbage Collector (GC) has to work overtime to clean them. GC pauses (brief moments where the app freezes while the Garbage Collector cleans up unused objects) slow down your app ‚Äî users experience random latency spikes (sudden delays between the user's click and the app's response). It's like buying a new notebook every time you need to jot one thing down, then immediately throwing it away.
One Object, Zero GC Pressure
The DI container (the framework's built-in system that automatically creates and connects objects for you) creates one ConfigService instance and reuses it for every request. One dictionary lives in memory for the app's entire lifetime ‚Äî no repeated allocations (reserving chunks of memory for new objects), no GC pressure (extra cleanup work for the Garbage Collector), no latency spikes. It's like having one notebook on your desk that you refer to every time.
Untestable
The junior's constructor does File.ReadAllText("appsettings.json") directly. In a unit test, that file probably doesn't exist ‚Äî the test crashes. Even if you copy the file, you can't control what's in it per-test. You're stuck testing with real files instead of fake data. This makes tests slow, brittle (tests that break for reasons unrelated to your code), and dependent on the environment.
Mock It, Test Anything
The senior codes against an interface (writes code that depends on a contract like IConfigService, not a specific class) ‚Äî not the concrete class (the actual implementation, not an interface). In tests, you inject (pass in through the constructor) a Mock<IConfigService> (a fake version of a dependency that you control in tests) that returns whatever values you want ‚Äî no files needed. Each test can set different config values, run in isolation (each test is self-contained and doesn't affect others), and execute in milliseconds. This is the power of Dependency Injection (letting a framework supply your objects instead of creating them yourself) + interfaces.
Violates DRY (Don't Repeat Yourself) + SRP (Single Responsibility Principle ‚Äî one class, one job)
Every service that needs config has to:
  • Know the concrete class (the actual implementation, not an interface) name ConfigService.
  • Call new ConfigService() itself.
  • Handle missing keys on its own.

Config-loading logic is copy-pasted everywhere. Worse, OrderService now has two jobs ‚Äî processing orders AND loading configuration. Rename the config class? Every file that uses it breaks.

One Registration, Clean Separation
  • Register (tell the framework: when someone asks for X, give them Y) once in Program.cs:
    builder.Services.AddSingleton<IConfigService, ConfigService>()
  • Every service just asks for IConfigService in its constructor ‚Äî never creates anything.
  • Switch to Azure App Configuration (a cloud service for storing config centrally)? Change one line. Zero other files touched.

Logging is everywhere in production. How logging is implemented reveals enormous differences in maturity, operability (how easy it is to run and maintain in a live system), and observability (your ability to understand what's happening inside the app by reading its outputs) mindset.

Why Junior Logging Fails ‚Äî 6 Problems

Completely Untestable
static LogHelper cannot be mocked (replaced with a fake version in tests) ‚Äî it's a concrete static class (a class with no interface that you call directly ‚Äî can't swap it out) baked into your code. In unit tests, every log call actually writes to Console, producing noise in test output. Worse, you can't verify "did my code log the right error message?" because there's no way to intercept (observe or redirect) static calls. It's like trying to test a phone call when the phone is bolted to the wall.
Inject ILogger<T> via DI
The senior uses ILogger<PaymentService> injected through constructor (passed in automatically by the framework). In tests, you pass a Mock<ILogger<PaymentService>> and verify exact log calls ‚Äî zero console noise, full control. The DI container (the framework's built-in system that automatically creates and connects objects for you) wires up (automatically creates and connects) the real Serilog (a popular third-party logging library for .NET) logger in production automatically.
No Structured Logging
String interpolation like $"Charging {customerId} for {amount}" produces flat text blobs (plain unstructured text with no searchable fields). In a log aggregator (a tool that collects all your logs in one place so you can search them) like Seq, Datadog, or Elastic, you can't query "show me all payments over $1,000 that failed" ‚Äî you'd have to write regex (complex pattern-matching expressions for searching text) through raw text. It's like writing receipts in paragraphs instead of a spreadsheet ‚Äî good luck searching later.
Message Templates with Named Properties
Serilog (a popular third-party logging library for .NET) uses message templates: "Charging customer {CustomerId} amount {Amount:C}". Each {placeholder} becomes a searchable, indexed field (pre-processed for fast lookup, like a database column). In Seq (a log viewer/search tool) you can query Amount > 1000 AND CustomerId = "C123" instantly. Structured logging (each log entry is a record with named fields instead of a plain line of text) turns logs from text files into queryable databases.
No Log Levels
Every message hits Console.WriteLine ‚Äî there's no way to show only errors in production but debug info in staging (a test environment that mirrors production). Under concurrent load, Console.WriteLine is synchronized (only one thread can write at a time ‚Äî all others wait), creating a bottleneck (the one slow point that limits the whole system's speed). Your logging literally slows down your app when you need it most.
MinimumLevel + Async Sinks
Serilog (a popular third-party logging library for .NET) uses log levels (categories like Debug, Info, Warning, Error that let you filter by importance) ‚Äî .MinimumLevel.Information() filters at the source so debug logs don't even get formatted in production. You can change levels at runtime via appsettings.json without redeploying. Sinks (destinations where logs get written ‚Äî console, file, Seq, etc.) like Seq use async batching (entries written on a background thread in groups, so your code never waits), so logging never blocks your request pipeline (the chain of code that processes a user's HTTP request).
Thread-Unsafe Console Colors
Console.ForegroundColor is global mutable state (a value stored in one shared place that any thread can change at any moment). Thread A sets it to Red, Thread B sets it to Blue, Thread A resets ‚Äî you get garbled rainbow output. Under high concurrency (many threads running at the same time), error messages might appear in green and info messages in red. The colors become meaningless noise.
JSON + External Viewers
The senior outputs logs as JSON via new JsonFormatter(). No console colors needed ‚Äî log viewers like Seq (a log viewer/search tool) and Kibana (a log visualization tool) display colors based on log level automatically. Each log entry is a self-contained JSON object, so there's zero shared mutable state and zero thread-safety (no risk of threads corrupting each other's data) issues.
Tight Coupling
Tight Coupling (code so deeply linked to a specific class that swapping it out requires changing many files) ‚Äî every class calls LogHelper.Log() directly, forming a concrete static dependency (hardwired to one specific class with no way to swap it). Want to switch from Console to Serilog (a popular third-party logging library for .NET), or add Application Insights (Microsoft's cloud monitoring service)? You'd have to find-and-replace across every file in the codebase. It's like hardcoding a phone number instead of using a contact list ‚Äî change the number, update everywhere.
One Registration, Swap Anytime
Services depend on ILogger<T> ‚Äî an abstraction (a general contract like an interface, not tied to a specific implementation). The logging provider (the system that decides where logs actually go) is configured once at startup: builder.Host.UseSerilog(). Switch to NLog (another popular logging library)? Change one line. Add Application Insights (Microsoft's cloud monitoring service) alongside Serilog? Add one more sink. Zero service code changes.
No Correlation / Enrichment
When a failing request generates 20 log lines across 5 services, which lines belong together? The junior's logs have no correlation ID (a unique tag attached to every log line from one request, so you can filter by it), no request context (metadata like who made the request, what URL they hit), and no user identity. Debugging production issues means scrolling through thousands of interleaved (randomly mixed together in time order, not grouped by request) log lines hoping timestamps line up.
Automatic Enrichment Pipeline
Enrichment (automatically adding extra fields like correlation ID, machine name to every log entry) is set up once: .Enrich.WithCorrelationId() automatically tags every log entry with the HTTP request's correlation ID (a unique tag attached to every log line from one request, so you can filter by it). Add .Enrich.WithProperty("Application", "PaymentAPI") for service name. In Seq (a log viewer/search tool), filter by CorrelationId = "abc-123" to see the entire request journey across all services instantly.

Core Implementations

Implementation 1 ‚Äî Naive Singleton (BROKEN ‚Äî Race Condition)

The naive implementation is the first thing most developers reach for. It fails catastrophically under concurrent access.

NaiveSingleton.cs ‚Äî DO NOT USE IN PRODUCTION
public class NaiveSingleton
{
    private static NaiveSingleton? _instance;

    // Simulates expensive initialization (DB connection, ML model, etc.)
    private NaiveSingleton() { Thread.Sleep(100); }

    public static NaiveSingleton Instance
    {
        get
        {
            // RACE CONDITION:
            // Thread A and Thread B can BOTH see _instance == null
            // simultaneously, before either has assigned it.
            // Result: TWO instances created. Pattern violated.
            if (_instance == null)
                _instance = new NaiveSingleton();

            return _instance;
        }
    }
}

// Demonstration of the race condition:
var tasks = Enumerable.Range(0, 10)
    .Select(_ => Task.Run(() => NaiveSingleton.Instance));
var instances = await Task.WhenAll(tasks);
var uniqueCount = instances.Distinct().Count();
// uniqueCount could be 2, 3, or more ‚Äî NOT 1!
The Race Condition Explained
Thread A reads _instance == null as true. Before Thread A executes new NaiveSingleton(), Thread B also reads _instance == null as still true (Thread A hasn't written yet). Both threads proceed to construct. Both assign to _instance. In the end, one assignment wins ‚Äî but two constructors ran, two objects were created, and any initialization side effects (opening a DB connection, loading a model, starting a timer) happened twice.
Implementation 2 ‚Äî Double-Checked Locking (Thread-Safe)

The classic thread-safe approach. Use when you need lazy initialization with full control over the locking mechanism, or in environments without Lazy<T>.

ThreadSafeSingleton.cs
// sealed: prevents subclassing (subclass could create second instance)
public sealed class ThreadSafeSingleton
{
    // volatile: prevents CPU/JIT from reordering reads/writes
    // Without volatile, Thread B could see a non-null pointer
    // before the constructor has finished running
    private static volatile ThreadSafeSingleton? _instance;

    // Dedicated lock object ‚Äî never lock on 'this' or typeof(T)
    private static readonly object _lock = new();

    private ThreadSafeSingleton() { }

    public static ThreadSafeSingleton Instance
    {
        get
        {
            // First check: avoid acquiring lock on every call
            // (lock only needed during first initialization)
            if (_instance == null)
            {
                lock (_lock)
                {
                    // Second check: another thread may have
                    // initialized between our first check and
                    // acquiring the lock
                    if (_instance == null)
                        _instance = new ThreadSafeSingleton();
                }
            }
            return _instance;
        }
    }
}

The modern, idiomatic C# approach. Lazy<T> encapsulates all thread-safety logic in BCL-tested, production-hardened code ‚Äî no manual locks needed.

public sealed class ModernSingleton
{
    // LazyThreadSafetyMode.ExecutionAndPublication:
    // - Only ONE thread calls the factory lambda
    // - Other threads block until initialization completes
    // - All threads then see the same, fully-initialized instance

        new(() => new ModernSingleton(),
            LazyThreadSafetyMode.ExecutionAndPublication);

    private ModernSingleton() { }

    public static ModernSingleton Instance => _lazy.Value;

    // IsValueCreated: useful for diagnostics and health checks

}

4 Reasons Lazy<T> is Best for Hand-Rolled Singletons

BCL-Tested Thread Safety
The .NET BCL team has battle-tested Lazy<T> across millions of applications. Its double-checked locking is implemented with precise memory barrier semantics ‚Äî more reliable than hand-rolling it.
Exception Handling Built In
With ExecutionAndPublication, if the factory throws, the exception is cached and re-thrown on every subsequent access ‚Äî ensuring consistent failure behavior rather than silently swallowing errors on retry.
Diagnostics via IsValueCreated
You can check _lazy.IsValueCreated in health check endpoints to verify the Singleton was initialized without forcing initialization. Double-checked locking has no equivalent diagnostic property.
Communicates Intent Clearly
new Lazy<T>(() => new T()) communicates "intentionally lazy" to every reader. Double-checked locking is 10+ lines that require careful review to verify correctness each time.

Bug #1 ‚Äî Captive Dependency (Singleton holding Scoped DbContext)
Business Impact: $23,000 in lost transactions
An e-commerce Singleton OrderProcessor captured a Scoped DbContext at construction time. Because the Singleton lives in the root scope (which is never disposed), the captured DbContext was never disposed either ‚Äî it stayed alive but became stale. Its change tracker accumulated entities across requests, queries returned outdated cached data, and connections were never returned to the pool. The bug ran undetected for 4 hours during a weekend sale event until the connection pool was exhausted and all new requests started timing out.

public class ProductCacheService : IProductCacheService
{
    private readonly IMemoryCache _cache;
    private readonly ILogger<ProductCacheService> _logger;

    public ProductCacheService(
        IMemoryCache cache,
        ILogger<ProductCacheService> logger)
    {
        _cache = cache;
        _logger = logger;
    }

    public async Task<Product?> GetOrAddAsync(
        string cacheKey,
        Func<Task<Product?>> factory,
        int cacheSizeUnits = 1)
    {
        if (_cache.TryGetValue(cacheKey, out Product? cached))
            return cached;

        var product = await factory();

        if (product is not null)
        {
            var options = new MemoryCacheEntryOptions
            {
                // Sliding expiration: entry expires if not accessed
                SlidingExpiration = TimeSpan.FromMinutes(15),
                // Absolute expiration: entry ALWAYS expires after this
                AbsoluteExpirationRelativeToNow = TimeSpan.FromHours(1),
                // Size: counted against SizeLimit
                Size = cacheSizeUnits,
                // Low-priority entries evicted first when under pressure
                Priority = CacheItemPriority.Low
            };

            _cache.Set(cacheKey, product, options);
            _logger.LogDebug("Cache SET: {Key}", cacheKey);
        }

        return product;
    }
}

S ‚Äî Single Responsibility Principle

A God Singleton does everything: logging, config, caching, metrics. The SRP says each class should have one reason to change. A God Singleton has as many reasons to change as it has responsibilities.

BEFORE ‚Äî God Singleton (SRP Violation)

// caching, config, AND metrics reasons

        new(() => new AppServices());

    // Logging
    public void Log(string msg) => Console.WriteLine(msg);

    // Config
    private readonly Dictionary<string, string> _config = new();
    public string GetSetting(string key) => _config[key];

    // Cache
    private Dictionary<string, object> _cache = new();
    public object? Get(string key) =>
        _cache.TryGetValue(key, out var v) ? v : null;

    // Metrics
    public void RecordMetric(string name, double value) { }
}

O ‚Äî Open/Closed Principle

A concrete static Singleton is sealed against extension. You cannot add new behaviors without modifying the class. With DI and interfaces, you extend behavior by adding new implementations ‚Äî not by modifying existing code.

Senior Decision Rule

Prefer DI Singleton over hand-rolled Singleton. Prefer Scoped over Singleton when in doubt. Use static class ONLY for pure functions with no state.

Singleton vs Static Class ‚Äî When to Use Which

Use Singleton When

  • You need to implement an interface for testability
  • You need lazy initialization (expensive resource, load on first use)
  • You need to be injected via DI into constructors
  • The service needs dependencies injected into it (ILogger, IConfiguration)
  • You need polymorphism ‚Äî ability to swap implementations
  • The class might need to be mocked in tests

Use Static Class When

  • The class is purely stateless ‚Äî pure functions only (Math, StringUtils)
  • You never need to inject it ‚Äî callers call it directly
  • No interface needed ‚Äî callers depend on concrete implementation
  • Performance is critical ‚Äî static calls are slightly faster (no vtable lookup)
  • The utility methods are trivially testable by calling them directly
  • The class will never have mutable state ‚Äî guaranteed by design
The Monostate Anti-Pattern
Monostate uses multiple instances that all share the same static state. It appears like regular classes (you use new Monostate()) but behaves like a Singleton. It combines the worst of both worlds: the complexity of instance management WITH shared global state. Avoid it ‚Äî use a proper Singleton via DI instead.

Note: Static fields are excluded from JSON/XML serialization ‚Äî serializing a Monostate instance produces an empty object, which can cause subtle data-loss bugs.

Q1. What is the Singleton pattern and when should you use it? Conceptual
Follow-up: What are the two problems it solves?
How to think about it: Don't just memorize the textbook definition. Instead, tell a story: "Here's the problem I faced, and here's how Singleton solved it." Ask yourself: what goes wrong if you create TWO of this thing? Walk through that problem first, then explain how Singleton fixes it. Think about it like this ‚Äî imagine you have a school where students are constantly asking "who is the principal?" If you could create multiple principals, the school would be chaos. Singleton is the rule that says: there can only be ONE principal, and everyone who asks gets the same person.

Answer: The Singleton pattern makes sure only ONE object of a class ever exists in your entire application ‚Äî like having only one principal in a school. No matter who asks for it, they all get the same one. It solves two problems at once: (1) it stops your code from accidentally creating duplicate objects ‚Äî because some objects MUST be unique, and (2) it makes that single object easy to reach from anywhere in your code, without having to pass it through every method call like a baton in a relay race.

When to use it: Use Singleton when having TWO copies of something would cause problems. For example: a connection pool (a collection of open database connections shared by everyone) ‚Äî if you had two pools, one would be empty while the other is overloaded. An in-memory cache ‚Äî if you had two caches, updating one wouldn't update the other, so different parts of your app see different data. A logger, a circuit breaker, a feature flag reader. The test is simple: "would two separate instances of this class cause bugs?"

The second is the "DI Singleton" ‚Äî where you just tell the framework (AddSingleton) "create this class once and reuse it forever." They're different tools for different problems. The hand-rolled version is a structural decision (the class itself enforces one-instance-only). The DI version is a lifetime decision ‚Äî meaning: the framework decides when to create the object and how long to keep it alive, not the class itself. Great candidates also mention: Singletons can be controversial because they create global shared state ‚Äî and shared state means any bug in one part of your app can affect every other part. Q2. Explain the four approaches to thread-safe Singleton in C# and their tradeoffs. Conceptual
Follow-up: Which would you use in production and why?
How to think about it: Walk through this as a journey from "broken but obvious" to "correct and production-ready." Think of it like building a better lock for a bathroom ‚Äî first attempt has no lock at all (disaster), second attempt always locks even when no one is in (slow), third attempt only locks when someone might be entering (smart). Your job is to explain each step and why it's an improvement over the previous one.

Answer:

1. Naive (broken): No protection at all. Imagine two people simultaneously checking if a bathroom is empty ‚Äî both see "empty" at the exact same moment and both walk in. In code, two threads can both check _instance == null at the same time and both create a new instance ‚Äî so you end up with two Singletons. This defeats the whole point.

2. Lock-Every-Read (safe but slow): Put a lock around the entire getter. A lock ‚Äî like a key to a room ‚Äî means only one thread can enter at a time. This is safe, but every single read (even after the object is already created) has to wait for the lock. Under heavy traffic, all threads queue up at the door, one by one. Correct, but slow.

3. Double-Checked Locking (fast, but needs volatile): Check if the object exists BEFORE locking. If it already exists, skip the lock entirely. Only lock when creating for the first time. This is fast, but there's a subtle trap: CPUs and compilers can reorder instructions for speed. Without marking the field as volatile ‚Äî which means "always read this fresh from memory, never from CPU cache" ‚Äî one thread might see the variable as non-null before the constructor has actually finished running. Modern CPUs and compilers sometimes rearrange the order your code runs in ‚Äî as an optimization. This is normally invisible, but in multi-threaded code it can cause one thread to see a variable in an incomplete state. This is a hardware-level timing issue, not a code bug. The volatile keyword adds a memory barrier ‚Äî a guarantee that all writes before it are visible to all threads after it.

4. Lazy<T> with ExecutionAndPublication (recommended for hand-rolled): Lazy<T> is a built-in C# type that does "create this object only when first needed, and do it safely." Think of it as a sealed envelope ‚Äî the letter inside is only written when you first open it, and if two people try to open it at once, only one writes the letter and the other waits. The ExecutionAndPublication mode means: only ONE thread creates the object, all others wait. Simpler, battle-tested, no need to worry about volatile.

5. DI Container (production standard): Let the framework handle it. When you call AddSingleton, ASP.NET Core creates the object once and gives everyone the same one. Thread-safe, testable, and you can swap implementations without touching any other code.

What separates good from great: Great candidates can name the exact failure mode: it is called the 'partially constructed object' bug ‚Äî one thread sees the reference as non-null but the object's fields haven't been assigned yet. This is distinct from a race condition creating two instances. Most developers just copy-paste double-checked locking without understanding why volatile is there. Knowing this shows real depth.
Q3. "Singleton is an anti-pattern" ‚Äî agree or disagree? Defend your position. Conceptual
Follow-up: When is it definitely an anti-pattern?
How to think about it: Don't take a hard yes-or-no stance. The honest answer is: "It depends on HOW you use it." This kind of nuanced answer impresses interviewers far more than a one-sided opinion. Think of it like a kitchen knife ‚Äî a knife is not an anti-pattern. But using a kitchen knife as a screwdriver is. The same pattern can be used well or badly.

Answer ‚Äî It Depends on Implementation:

When Singleton IS an anti-pattern: (1) Hand-rolled static access via MySingleton.Instance ‚Äî this is like hardcoding a phone number instead of looking it up. You can never swap it out for a fake version in tests, the dependency is hidden inside the class, and it creates global shared state that any code can mess with. (2) God Singleton ‚Äî one class that does everything (logging, config, cache, metrics) ‚Äî this violates the Single Responsibility Principle: every class should have ONE job. (3) Singleton holding a shorter-lived dependency ‚Äî a Singleton holding a database connection that was meant to be created fresh per request. This is like a permanent employee borrowing a visitor badge indefinitely. (4) An unbounded cache ‚Äî a Singleton that stores data forever with no cleanup, eventually eating all available memory.

When Singleton is NOT an anti-pattern: (1) Registered through the DI framework with an interface ‚Äî AddSingleton<ICache, MemoryCache>(). You can replace it with a fake in tests, inject it normally, and swap implementations without touching anything else. Fully testable and clean. (2) Immutable configuration registered once at startup ‚Äî if nothing changes it after creation, there are no threading problems or shared-state bugs. (3) Thread-safe, bounded shared resources ‚Äî like a connection pool with a fixed size, or a circuit breaker that tracks failure counts globally.

What separates good from great: Making the distinction between two different things that both use the word "Singleton": the structural pattern (the class itself controls how many instances exist) vs lifetime management (the framework decides to create this once and reuse it). When you use AddSingleton in ASP.NET Core, the CLASS doesn't need to do anything special ‚Äî the framework just creates it once and hands out the same object every time someone asks. That means you can have a Singleton-lifetime service that is fully testable, has an interface, and can be swapped out in tests ‚Äî none of the traditional "anti-pattern" problems apply. That insight is what separates someone who just memorized the pattern from someone who really understands it.
Q4. Explain the three LazyThreadSafetyMode values and when to use each. Conceptual
Follow-up: What happens if the factory throws?
How to think about it: Think of Lazy<T> as a "create on demand" wrapper. The three modes answer the question: "what happens if two threads try to create the object at the same time?" Each mode makes a different choice about safety vs speed. Then explain what happens when the creation fails ‚Äî most candidates skip this, but it reveals real production awareness.
LazyModes.cs
// Mode 1: ExecutionAndPublication (default, recommended for Singleton)
// - Only ONE thread executes the factory
// - Other threads block until initialization completes
// - If factory throws: exception is CACHED, re-thrown on every access
// - Use for: Singletons, expensive initialization, correctness required
var singleton = new Lazy<HeavyService>(
    () => new HeavyService(),
    LazyThreadSafetyMode.ExecutionAndPublication);

// Mode 2: PublicationOnly
// - MULTIPLE threads may execute the factory concurrently
// - Only the FIRST result is published; others are discarded
// - If factory throws: exception is NOT cached; next thread retries
// - Use for: cheap factory, idempotent init, no blocking wanted
var concurrent = new Lazy<LightService>(
    () => new LightService(),
    LazyThreadSafetyMode.PublicationOnly);

// Mode 3: None (no thread safety)
// - Factory runs once with NO synchronization
// - Use ONLY in single-threaded contexts (unit tests, etc.)
// - Fastest, but race condition possible in multi-threaded scenarios
var singleThread = new Lazy<Service>(
    () => new Service(),
    LazyThreadSafetyMode.None);
What separates good from great: The sneaky difference that trips everyone up ‚Äî what happens when the creation fails (throws an exception). With ExecutionAndPublication: the exception is remembered ("cached") forever. Every future attempt to access .Value throws the same exception again. The factory code never runs again. If it failed once, it's failed forever ‚Äî until you restart the app. With PublicationOnly: the exception is NOT remembered. The next thread that tries will run the factory fresh. This matters when your failure is temporary ‚Äî like trying to connect to a database that was momentarily unavailable. With ExecutionAndPublication, one bad connection attempt at startup permanently breaks the app. With PublicationOnly, the next request gets another chance to connect.
Q5. How does ASP.NET Core's DI container implement AddSingleton internally? Conceptual
Follow-up: What is the root scope and why does it matter?
How to think about it: This question tests whether you know what happens behind the scenes when you call AddSingleton. Think of the DI container as a registry ‚Äî like a hotel reception desk. When you register a Singleton, you're telling the desk: "if anyone asks for Room 42, give them the key to the same room every time." Then explain why the "root scope" (the master reception desk that never closes) causes a nasty bug when Singletons borrow short-lived objects.

Answer: When you call builder.Services.AddSingleton<IService, Service>(), the DI framework writes down a note called a ServiceDescriptor ‚Äî basically a recipe card that says: "to make an IService, create a Service object, and the lifetime is Singleton." When the app builds (via builder.Build()), the framework creates the master container ‚Äî the ServiceProvider ‚Äî available as app.Services. Think of it as the hotel's root reception desk that runs for the entire life of the application.

The root ServiceProvider keeps a lookup table: "service type ‚Üí the one object we created." The first time someone asks for the Singleton, the container creates it, stores it in this table, and from then on returns the same stored object to everyone who asks ‚Äî no matter where in the app they're asking from.

Each HTTP request creates a temporary child container called an IServiceScope ‚Äî like a temporary guest desk for that one request. Services registered as "Scoped" (meaning "one per request") are stored in this temporary desk and cleaned up when the request ends. Singleton services are NEVER stored in the temporary desk ‚Äî they always come from the master root desk, which never closes.

What separates good from great: Understanding the "captive dependency" bug that comes from this design. Imagine a Singleton service needs a database connection (which should be created fresh per request ‚Äî a Scoped service). When the DI container creates the Singleton for the first time, it looks up the database connection from the ROOT desk ‚Äî not the request desk. That database connection is then held by the Singleton permanently, never cleaned up, potentially holding an open transaction for the entire lifetime of the app. This is called a captive dependency ‚Äî the long-lived Singleton has "captured" a short-lived dependency and is keeping it alive far longer than intended. The setting ValidateScopes = true makes ASP.NET Core check for this mistake at startup and throw an error immediately, instead of letting the bug quietly corrupt data in production.

Q6. Design a thread-safe Feature Flag service as a Singleton. It must support real-time flag updates without restart. Design

How to think about it: A feature flag is basically a global on/off switch for parts of your app ‚Äî like "is the dark mode feature enabled?" public sealed class FeatureFlagService : IHostedService, IDisposable
{
    // Immutable snapshot ‚Äî replaced atomically via Interlocked
    private volatile IReadOnlyDictionary<string, bool> _flags
        = new Dictionary<string, bool>();

    private readonly IFlagProvider _provider; // IFlagProvider: your remote flag source (LaunchDarkly, database, etc.)
    private readonly ILogger<FeatureFlagService> _logger;
    private Timer? _refreshTimer;

    public FeatureFlagService(
        IFlagProvider provider,
        ILogger<FeatureFlagService> logger)
    {
        _provider = provider;
        _logger = logger;
    }

    // Thread-safe read ‚Äî no lock needed (volatile reference swap)
    public bool IsEnabled(string flagName)
        => _flags.TryGetValue(flagName, out var val) && val;

    public Task StartAsync(CancellationToken ct)
    {
        // Initial load
        _ = RefreshAsync();
        // Poll every 30 seconds
        _refreshTimer = new Timer(_ => _ = RefreshAsync(),
            null, TimeSpan.FromSeconds(30), TimeSpan.FromSeconds(30));
        return Task.CompletedTask;
    }

    private async Task RefreshAsync()
    {
        try
        {
            var newFlags = await _provider.GetAllFlagsAsync();
            // Volatile write with memory barrier ‚Äî readers always see a complete snapshot
            // The volatile keyword on _flags ensures this write is visible to all threads
            _flags = newFlags;
            _logger.LogDebug("Feature flags refreshed: {Count}", newFlags.Count);
        }
        catch (Exception ex)
        {
            // Keep serving old flags ‚Äî don't crash on refresh failure
            _logger.LogWarning(ex, "Feature flag refresh failed. Serving cached flags.");
        }
    }

    public Task StopAsync(CancellationToken ct) { _refreshTimer?.Change(Timeout.Infinite, 0); return Task.CompletedTask; }
    public void Dispose() => _refreshTimer?.Dispose();
}

What separates good from great: The clever trick in this design is replacing the entire dictionary at once instead of updating it entry by entry. Think of it like swapping out a whole menu board at a restaurant rather than changing individual items while customers are reading it. If you changed items one by one, a customer might see "Burger: $5" and then glance up a moment later and see "Burger: $8" ‚Äî a confusing mix of old and new. By swapping the entire board atomically (all at once, instantly), every reader either sees the old complete menu or the new complete menu ‚Äî never a half-updated mess. Q7. Design a Singleton-based rate limiter that allows N requests per second per API key. Design

How to think about it: A rate limiter is like a bouncer at a club ‚Äî it tracks how many people came in recently and stops letting more in if the limit is hit. A Singleton is the right choice because the counter must be shared across ALL requests (if each request had its own counter starting at zero, the limit would never be reached). RateLimiterService.cs
public sealed class InProcessRateLimiter : IRateLimiter
{
    // Token bucket per API key
    private readonly ConcurrentDictionary<string, SlidingWindowCounter> _counters = new();
    private readonly int _requestsPerSecond;

    public InProcessRateLimiter(int requestsPerSecond = 100)
        => _requestsPerSecond = requestsPerSecond;

    public bool TryConsume(string apiKey)
    {
        var counter = _counters.GetOrAdd(
            apiKey,
            _ => new SlidingWindowCounter(_requestsPerSecond));

        return counter.TryIncrement();
    }
}

internal class SlidingWindowCounter
{
    private readonly Queue<long> _timestamps = new();
    private readonly object _lock = new();
    private readonly int _limit;

    public SlidingWindowCounter(int limit) => _limit = limit;

    public bool TryIncrement()
    {
        var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        var windowStart = now - 1000; // 1 second window

        lock (_lock)
        {
            // Remove timestamps outside the window
            while (_timestamps.Count > 0 && _timestamps.Peek() < windowStart)
                _timestamps.Dequeue();

            if (_timestamps.Count >= _limit)
                return false; // Rate limit exceeded

            _timestamps.Enqueue(now);
            return true;
        }
    }
}

// Registration
builder.Services.AddSingleton<IRateLimiter, InProcessRateLimiter>();

The fix: use Redis (a fast shared database) to store the counters. All servers read and write to the same Redis counter, so the limit is enforced globally across all servers. Q8. Design a Singleton background job scheduler that can run jobs on a cron schedule. Design

How to think about it: A background job scheduler is a task that keeps running as long as the app is alive ‚Äî cleaning up old records, sending reminder emails, etc. Since it runs in the background for the app's lifetime, Singleton is the right fit. The tricky part is: what if the job is still running when its next scheduled time arrives? Like a train that hasn't arrived at the station yet when the next train is supposed to depart. You need a way to say "skip this run if the previous one isn't done yet." Focus on that overlap prevention as your key insight.

BackgroundService is a built-in .NET base class for tasks that run continuously in the background for the lifetime of the app ‚Äî like a loop that keeps processing work.

JobScheduler.cs
public class OrderCleanupJob : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    private readonly ILogger<OrderCleanupJob> _logger;
    // Prevents overlapping execution
    private readonly SemaphoreSlim _mutex = new(1, 1);

    public OrderCleanupJob(
        IServiceScopeFactory scopeFactory,
        ILogger<OrderCleanupJob> logger)
    {
        _scopeFactory = scopeFactory;
        _logger = logger;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            await Task.Delay(TimeSpan.FromMinutes(5), stoppingToken);

            // Non-blocking: skip this tick if previous run still in progress
            if (!await _mutex.WaitAsync(0, stoppingToken))
            {
                _logger.LogWarning("Job still running. Skipping this tick.");
                continue;
            }

            try
            {
                // Create a new scope for each job run (allows Scoped dependencies)
                using var scope = _scopeFactory.CreateScope();
                var orderRepo = scope.ServiceProvider
                    .GetRequiredService<IOrderRepository>();

                await orderRepo.DeleteExpiredOrdersAsync(stoppingToken);
                _logger.LogInformation("Order cleanup completed.");
            }
            catch (Exception ex)
            {
                _logger.LogError(ex, "Order cleanup failed.");
            }
            finally
            {
                _mutex.Release();
            }
        }
    }
}

Q9. Design a Singleton connection pool that handles connection failures gracefully. Design
Follow-up: How do you handle connection validation before lending?
How to think about it: A connection pool is like a library that lends out books ‚Äî except the "books" are database connections, which are expensive to create (opening a connection takes time and resources). Instead of creating a brand-new connection for every request, we keep a collection of reusable connections. Think of it as a car rental agency: you pick up a car when you need it, return it when done, and someone else can use it next. Your design needs to handle: (1) checking the "car" still runs before handing it over, (2) getting a replacement if a car breaks down, (3) telling customers to wait if all cars are out (not failing immediately), and (4) enforcing a maximum fleet size so the agency doesn't rent out more cars than it owns.
ConnectionPool.cs
public sealed class ManagedConnectionPool
{
    private static readonly Lazy<ManagedConnectionPool> _lazy =
        new(() => new ManagedConnectionPool());
    public static ManagedConnectionPool Instance => _lazy.Value;

    private readonly ConcurrentBag<IDbConnection> _pool = new();
    private readonly SemaphoreSlim _throttle;
    private readonly string _connectionString;
    private const int MaxPoolSize = 20;
    private const int MinPoolSize = 5;

    private ManagedConnectionPool()
    {
        _connectionString = Environment.GetEnvironmentVariable("DB_CONN")!;
        _throttle = new SemaphoreSlim(MaxPoolSize, MaxPoolSize);

        // Pre-warm minimum connections
        for (int i = 0; i < MinPoolSize; i++)
            _pool.Add(CreateValidConnection());
    }

    public async Task<PooledConnection> RentAsync(
        TimeSpan timeout = default)
    {
        var effectiveTimeout = timeout == default
            ? TimeSpan.FromSeconds(30) : timeout;

        // Block if pool is at max capacity
        if (!await _throttle.WaitAsync(effectiveTimeout))
            throw new TimeoutException(
                $"Connection pool exhausted. Waited {effectiveTimeout}.");

        // Try to get a valid connection from pool
        while (_pool.TryTake(out var conn))
        {
            if (IsConnectionAlive(conn))
                return new PooledConnection(conn, this);

            // Dead connection ‚Äî discard and let throttle count stand
            // (we'll create a new one implicitly)
            conn.Dispose();
        }

        // No pooled connections available ‚Äî create new (within throttle)
        return new PooledConnection(CreateValidConnection(), this);
    }

    internal void Return(IDbConnection conn)
    {
        if (IsConnectionAlive(conn))
            _pool.Add(conn);
        else
            conn.Dispose(); // Discard dead connections

        _throttle.Release(); // Always release the slot
    }

    private static bool IsConnectionAlive(IDbConnection conn)
    {
        try
        {
            if (conn.State != ConnectionState.Open) return false;
            using var cmd = conn.CreateCommand();
            cmd.CommandText = "SELECT 1";
            cmd.ExecuteScalar();
            return true;
        }
        catch { return false; }
    }

    private IDbConnection CreateValidConnection()
    {
        var conn = new SqlConnection(_connectionString);
        conn.Open();
        return conn;
    }
}

// RAII wrapper: auto-returns connection when disposed
public sealed class PooledConnection : IAsyncDisposable
{
    private readonly IDbConnection _conn;
    private readonly ManagedConnectionPool _pool;
    public IDbConnection Connection => _conn;

    internal PooledConnection(IDbConnection conn, ManagedConnectionPool pool)
        => (_conn, _pool) = (conn, pool);

    public ValueTask DisposeAsync()
    {
        _pool.Return(_conn);
        return ValueTask.CompletedTask;
    }
}
What separates good from great: Two things stand out. First: SemaphoreSlim ‚Äî think of it as a ticket machine at a deli counter. If you set it to 20, only 20 people can have a ticket at once. When someone new arrives, they wait until a ticket is returned. This is called "backpressure" ‚Äî instead of failing with an error immediately when the pool is full, we make callers wait. That's usually the right behavior. Second: the PooledConnection RAII pattern ‚Äî RAII stands for 'Resource Acquisition Is Initialization' ‚Äî a pattern where cleanup (returning the connection) is tied to the object's disposal. In practice: when this wrapper object is disposed (thrown away), the connection automatically returns to the pool. The await using keyword guarantees cleanup happens even if an exception is thrown. This prevents connections from being permanently lost. Also worth saying: in real production code, SqlClient already has built-in connection pooling ‚Äî you don't need to build this yourself. This design exercise is to show you understand the mechanics behind it.
Q10. Why must an HttpClient be a Singleton (via IHttpClientFactory) rather than created per-request? Design
Follow-up: What is socket exhaustion and how does IHttpClientFactory prevent it?
How to think about it: This is one of the most famous "gotcha" bugs in .NET ‚Äî developers hit this in production all the time without understanding why. Start by explaining what happens at the networking level when you create a new HttpClient per request. Then explain why the naive fix (making HttpClient a static field) also has a problem. Finally explain how IHttpClientFactory solves BOTH problems at once.

Answer: HttpClient internally holds something called an HttpMessageHandler, which manages a pool of network connections (called TCP sockets ‚Äî think of them as phone lines between your app and a remote server). When you call new HttpClient() and then throw it away (dispose it), those phone lines don't hang up immediately. They enter a state called TIME_WAIT ‚Äî the OS keeps them "on hold" for 2-4 minutes to make sure all data was received before closing. Now imagine 1,000 requests per second, each creating a new HttpClient. After 2 minutes, you have 120,000 "phone lines on hold." Your operating system has a hard limit on how many of these 'on-hold' connections it can track simultaneously ‚Äî typically in the tens of thousands. Once you hit that limit, new connections fail with "address already in use" ‚Äî your service stops working completely. This is called socket exhaustion.

IHttpClientFactory is a Singleton that maintains a shared pool of HttpMessageHandler instances ‚Äî it reuses the same underlying connections across many requests instead of creating new ones. Every 2 minutes it creates a fresh handler (so it picks up any DNS address changes) but keeps the old one alive long enough for in-flight requests to finish. This solves BOTH the socket exhaustion problem (handlers are shared and reused) and the DNS staleness problem (handlers are periodically refreshed to pick up updated IP addresses).

{
    client.BaseAddress = new Uri("https://api.stripe.com");
    client.DefaultRequestHeaders.Add("Authorization", "Bearer sk_...");
    client.Timeout = TimeSpan.FromSeconds(30);
})
.AddTransientHttpErrorPolicy(p => p.RetryAsync(3))          // Polly retry
.AddTransientHttpErrorPolicy(p => p.CircuitBreakerAsync(5, TimeSpan.FromMinutes(1)));

// Usage
public class StripeService

    public StripeService(IHttpClientFactory factory) => _factory = factory;

What separates good from great: Knowing the second trap: the naive fix "just make HttpClient a static Singleton field" avoids socket exhaustion, but creates a new problem called DNS staleness. Here's why: when your app first connects to a remote server (say, an API), it looks up the server's IP address using DNS (like checking a phone directory). If you reuse the same HttpClient forever, it caches that IP address and never looks it up again. But remote services change their IP addresses all the time ‚Äî for load balancing, blue/green deployments, failovers. If the IP changes and your HttpClient doesn't know, it keeps sending requests to an address that no longer works. IHttpClientFactory solves this by swapping out the handler every 2 minutes ‚Äî forcing a fresh DNS lookup while still reusing connections efficiently. It's the only approach that solves BOTH problems simultaneously.

Q11. Your Singleton service throws ObjectDisposedException after the first request. What's wrong and how do you fix it? Debug
Follow-up: How do you catch this at startup instead of runtime?
How to think about it: This is called the captive dependency problem ‚Äî 'captive' because a long-lived service has trapped (captured) a short-lived dependency and is keeping it alive far longer than intended. Think through the chain of events step by step: a permanent (Singleton) service holds a reference to a temporary (Scoped) service. The temporary service gets cleaned up (disposed) at the end of the first request. On the second request, the permanent service tries to use the already-cleaned-up temporary service ‚Äî and it explodes. Walk through this story clearly, then offer three ways to fix it.

Diagnosis: Your Singleton service has a constructor dependency on a Scoped service ‚Äî typically a DbContext (the EF Core database context), an IRepository, or similar. Here's what happens: when the DI container creates the Singleton for the first time, it grabs the Scoped service from the root container (the permanent one). This creates a mismatch ‚Äî the Scoped service was designed to be fresh per request and then thrown away, but now it's being held hostage by a Singleton that lives forever. When the Scoped service's internal resources (database connection, open transaction) get cleaned up after the first request ends, the Singleton is still holding a reference to the now-broken shell. On the second request, the Singleton tries to use it ‚Äî crash.

Fix 1: Use IDbContextFactory<T> ‚Äî instead of holding a DbContext directly, hold a factory that creates a fresh one on demand. Create it, use it, dispose it ‚Äî all within a single method call.

Fix 2: Use IServiceScopeFactory ‚Äî ask the framework to create a temporary scope (a mini-container for one operation), get your short-lived services from it, do your work, then throw the scope away. The background job example in Q8 shows exactly this pattern.

Fix 3: Change the service lifetime to Scoped (if it doesn't actually need to be a Singleton). Ask yourself: does this service truly need to be shared across ALL requests? If not, make it Scoped.

Prevention: Enable ValidateScopes = true in development ‚Äî this makes ASP.NET Core check for this mistake at startup and throw an immediate error, instead of letting the bug silently corrupt data in production:

Program.cs
builder.Host.UseDefaultServiceProvider(options =>
{
    options.ValidateScopes = builder.Environment.IsDevelopment();
    options.ValidateOnBuild = true;
});
What separates good from great: There are actually TWO ways this bug can manifest, and the second one is harder to find. The first way: the Scoped service gets fully cleaned up and then used ‚Äî you get an ObjectDisposedException right away. This is painful but at least obvious. The second way (the sneaky one): the Scoped service doesn't get cleaned up (because the root container holds it alive) but it accumulates wrong state over time ‚Äî a database transaction that should have been rolled back, an EF Core change tracker full of stale objects from the previous request. In this case there's NO exception ‚Äî your app just quietly returns wrong data. A customer might see someone else's order details, or a product update might silently fail. This second mode is far more dangerous because it's invisible until something is clearly wrong in production.
Q12. Your application is leaking memory. Profiling shows a Singleton service holding millions of objects. What are the likely causes and fixes? Debug
Follow-up: How do you monitor Singleton memory usage in production?
How to think about it: A Singleton lives for the entire lifetime of the app. Anything it holds onto also lives for the entire lifetime of the app. Memory leaks in Singletons happen when something keeps ADDING to a collection inside the Singleton but never REMOVES anything. Think of it like a sink with the drain blocked ‚Äî water keeps flowing in but can never drain out. Identify the three most common "blocked drains" in Singleton services.

Cause 1 ‚Äî Unbounded Dictionary (the most common cause): A cache keyed by user IDs, request IDs, URLs, or session IDs where new keys are constantly added but old ones are never removed. After a few days of traffic, this dictionary has millions of entries. Fix: Use IMemoryCache with a SizeLimit and sliding expiration ‚Äî old entries automatically get cleaned up after they haven't been accessed for a set time period.

Cause 2 ‚Äî Event Handler Leaks: Your Singleton registers itself to listen to events (like "user logged in" or "config changed") but never unregisters. In C#, when you subscribe to an event (like button.Click += MyHandler), the event owner (the publisher ‚Äî the thing firing the event) holds a reference to you (the subscriber ‚Äî the thing listening). As long as the event exists, the subscriber can never be garbage collected ‚Äî the GC (Garbage Collector ‚Äî the part of .NET that automatically frees unused memory) sees it as "still in use" because something is pointing at it. Fix: use a WeakEventManager (which holds a weak reference ‚Äî a reference that doesn't count as 'in use' from the garbage collector's perspective, so the object can still be cleaned up), or unsubscribe from the event when the service is disposed.

Cause 3 ‚Äî Growing Work Queue Without Draining: A Singleton that receives incoming work items into a queue, but the background worker processing them is too slow. Under high load, items pile up faster than they're processed. The queue grows forever until memory runs out. Fix: Use a bounded channel (BoundedChannel<T>) ‚Äî a queue with a maximum size. When it's full, either make new arrivals wait, or drop the oldest items to make room.

BoundedQueue.cs ‚Äî Fix for growing queue
// Replace unbounded Queue with bounded Channel
private readonly Channel<WorkItem> _channel =
    Channel.CreateBounded<WorkItem>(new BoundedChannelOptions(10_000)
    {
        FullMode = BoundedChannelFullMode.DropOldest, // Drop old items when full
        SingleReader = false,
        SingleWriter = false
    });
What separates good from great: Knowing how to diagnose a memory leak in production, not just in theory. Two command-line tools: dotnet-counters monitor --process-id PID System.Runtime ‚Äî this shows you live memory metrics (GC heap size, number of objects) as the app runs. If the heap is growing steadily and never coming back down, you have a leak. dotnet-gcdump collect ‚Äî captures a snapshot of everything currently in memory (a "heap dump"). You can load this into Visual Studio or PerfView and see which objects are using the most memory and what is holding onto them. Also a great answer: expose a health check endpoint that reports how many items are in your Singleton caches. Instead of waiting for an outage, you can alert when a cache grows above a threshold ‚Äî catching the leak before it becomes a crisis.
Q13. Your service creates two instances of a Singleton in production. How is this possible and how do you prevent it? Debug
Follow-up: Can reflection break a Singleton?
How to think about it: If you have two instances of a Singleton, that means there are two different code paths that each created one independently. Work through the possible ways that could happen: a race condition during creation, multiple separate registrations, or someone bypassing the rules using reflection. This question is also a good reminder that "Singleton" is a promise your code makes, not an absolute physical law ‚Äî it can be broken.

Cause 1 ‚Äî Race Condition in Naive Implementation: Two threads both check "is the instance created?" at the exact same moment. Both see "no." Both create a new instance. Now you have two. Fix: Use Lazy<T> with ExecutionAndPublication ‚Äî this guarantees only one thread runs the creation code.

Cause 2 ‚Äî Multiple AppDomains (old .NET Framework only): An AppDomain is like a separate isolated compartment inside one process. In the old .NET Framework, you could have multiple AppDomains in one app ‚Äî each would have its own copy of all static fields, including your Singleton. Each compartment thinks it has the one true Singleton, but there are actually multiple. Not a problem in modern .NET Core (which only ever has one AppDomain per process).

Cause 3 ‚Äî Multiple DI Registrations: If someone accidentally registers the same interface twice with different implementations, the DI container keeps both. When you call GetService<IService>(), you get the last one registered. But when you call GetServices<IService>() (plural), you get all of them ‚Äî multiple instances. Fix: don't double-register unless you intentionally want multiple implementations.

Cause 4 ‚Äî Reflection Bypass: Yes, reflection ‚Äî the C# feature that lets you inspect and call private members of a class ‚Äî can bypass a private constructor and create additional instances. It's like having a master key that can open any lock:

ReflectionBreak.cs
// Reflection CAN bypass private constructor
var type = typeof(MySingleton);
var ctor = type.GetConstructor(
    BindingFlags.Instance | BindingFlags.NonPublic,
    null, Type.EmptyTypes, null);
var second = ctor!.Invoke(null); // Creates a SECOND instance!

// Prevention: throw in constructor if instance already exists
private MySingleton()
{
    if (_lazy.IsValueCreated)
        throw new InvalidOperationException(
            "Use MySingleton.Instance ‚Äî do not use reflection.");
}
What separates good from great: In modern cloud deployments, "two instances of my Singleton" often doesn't mean two objects in the same process ‚Äî it means two completely separate server instances (called "pods" in Kubernetes). Each server runs its own copy of your app, with its own memory, its own Singleton, its own state. This is actually INTENDED ‚Äî it's how cloud apps handle more traffic by running multiple copies. The problem arises when those multiple Singletons hold state that should be consistent across ALL of them ‚Äî like a cache where you want everyone to see the same data, or a rate limiter that should count requests globally. The fix: move that shared state OUT of the in-process Singleton and INTO a shared external store like Redis, which all servers can read and write to.
Q14. What happens if the Singleton constructor throws an exception? How does the behavior differ between hand-rolled and DI approaches? Debug
Follow-up: How do you implement retry logic for Singleton initialization?
How to think about it: "What happens when creation fails?" is a practical question that catches many developers off guard. Think about it from the user's perspective: if the Singleton fails to initialize, can the app recover and try again, or is it permanently broken until someone restarts it? The answer depends on WHICH approach you used. Walk through each approach and explain its behavior when things go wrong.

Lazy<T> (ExecutionAndPublication mode ‚Äî the default): If the creation code throws an error, that error is permanently remembered. Every future attempt to access .Value throws the exact same error again ‚Äî forever, until the app restarts. The creation code is NEVER run again. This is like a vending machine that jams on the first use and stays jammed forever. Bad if your failure is temporary (like a database that was briefly unavailable at startup).

Lazy<T> (PublicationOnly mode): If the creation code throws, the error is NOT remembered. The next thread that tries will run the creation code fresh. This is like a vending machine that resets itself after a jam ‚Äî the next customer can try again. Use this mode when you expect the creation might fail temporarily and want automatic retry.

DI Container (AddSingleton): The DI container does NOT permanently cache failures. If the constructor throws, the exception propagates to whoever asked for the service, and the next request can try again. However, in ASP.NET Core, the Singleton is usually created at app startup ‚Äî if it throws there, it typically crashes the entire startup process, which means the app never starts accepting requests at all.

RetryInit.cs
// Retry pattern for Singleton initialization that may fail transiently
public sealed class DatabaseSingleton
{
    private static readonly Lazy<DatabaseSingleton> _lazy =
        new(CreateWithRetry, LazyThreadSafetyMode.ExecutionAndPublication);

    public static DatabaseSingleton Instance => _lazy.Value;

    private static DatabaseSingleton CreateWithRetry()
    {
        var retryPolicy = Policy
            .Handle<SqlException>()
            .WaitAndRetry(5, attempt =>
                TimeSpan.FromSeconds(Math.Pow(2, attempt)));

        return retryPolicy.Execute(() => new DatabaseSingleton());
    }

    private static readonly string _connectionString =
        Environment.GetEnvironmentVariable("DB_CONN")
        ?? throw new InvalidOperationException("DB_CONN not set");

    private DatabaseSingleton()
    {
        // May throw SqlException if DB is not yet ready
        using var conn = new SqlConnection(_connectionString);
        conn.Open(); // Throws if DB not ready
    }
}
What separates good from great: Instead of putting retry logic inside the constructor (which makes the constructor complex and hard to test), recommend separating the concerns: use an IHostedService to handle startup initialization with retry logic, and a health check endpoint to signal when the service is ready. Here's why this is better: while the service is still trying to initialize (retrying a failed database connection), the health check reports "Unhealthy." Kubernetes sees this and does NOT route any user traffic to this server until the health check says "Healthy." This means users never hit a server that isn't fully ready ‚Äî no failed requests, no errors, clean startup. The constructor stays simple; the retry logic is cleanly separated into its own place.
Q15. Your Singleton configuration service returns stale data after a config update. How do you implement live config reload? Debug
Follow-up: How do you handle mid-request config changes safely?
How to think about it: Config services are almost always Singletons ‚Äî they're created once at startup and shared everywhere. The problem: if the config changes (someone updates a setting file), a Singleton reads it once and never checks again. Think of it like a GPS that only downloads the map when you start your trip ‚Äî if a new road opens, your GPS doesn't know about it. .NET has three built-in ways to read config, each with different "freshness" behavior. Knowing the difference between them is what this question is really testing.
LiveConfigReload.cs
// IOptions<T>: Singleton ‚Äî reads config ONCE at startup. STALE after reload.
// IOptionsSnapshot<T>: Scoped ‚Äî reads current config per request. NOT usable in Singletons.
// IOptionsMonitor<T>: Singleton-safe ‚Äî CurrentValue always returns latest config.

{

        IOptionsMonitor<FeatureOptions> options,

        _options = options;
        _logger = logger;

        // OnChange callback signature: Action<TOptions, string>

        _options.OnChange((newOptions, name) =>

            _logger.LogInformation(
                "FeatureOptions changed. NewBeta={Beta}",
                newOptions.BetaEnabled);
        });
    }

    public bool IsBetaEnabled()

What separates good from great: Knowing the three options interfaces and when to use each: IOptions<T> ‚Äî reads config ONCE at startup and never changes. Use for config that should be fixed for the app's lifetime. IOptionsSnapshot<T> ‚Äî reads the CURRENT config once per HTTP request. Great for catching updates between requests. BUT it's a Scoped service, which means it cannot be injected into Singletons (captive dependency problem!). IOptionsMonitor<T> ‚Äî the Singleton-safe option. CurrentValue always gives you the latest config, and the OnChange callback lets you react immediately when something changes. Q18. Compare Singleton pattern vs Service Locator pattern. Why is Service Locator considered an anti-pattern? Compare
Follow-up: Is ServiceProvider.GetService<T>() a service locator?
How to think about it: The Service Locator pattern is like a class going to a warehouse and picking up whatever tools it needs, without telling anyone what it grabbed. Singleton via constructor injection is like listing everything you need on a requisition form ‚Äî everyone can see exactly what you depend on before you start work. The key difference is visibility: who can see what a class needs? This matters enormously for testing and understanding code.

Service Locator Anti-Pattern: A class calls a global registry inside its methods to fetch its own dependencies ‚Äî like ServiceLocator.GetService<ICache>(). Here's why this is bad: (1) Hidden dependencies ‚Äî you cannot tell what a class needs by looking at its constructor. You have to read through all its method bodies to discover what it secretly pulls from the locator. (2) Untestable ‚Äî you can't inject a fake version of ICache in tests because the class grabs it internally from the locator ‚Äî you'd have to configure the entire locator for every test. (3) Runtime failures ‚Äî if a service isn't registered in the locator, you get an error at runtime, not at compile time. Compile-time errors are caught instantly; runtime errors might only appear in production. (4) Action at a distance ‚Äî a term for when code in one place secretly changes behaviour somewhere completely different ‚Äî the class's behavior can change based on what's in the locator, which is invisible to whoever is calling the class.

(2) Middleware and filters that need to pick which service to use based on runtime conditions. (3) Factory methods that dynamically choose between multiple implementations. Outside these specific cases, always inject dependencies through the constructor where they're visible and testable.
Q19. How does Singleton combine with Factory Method pattern? Give a real example. Compare

How to think about it: The Factory Method pattern is about choosing WHICH object to create based on some input ‚Äî like a vending machine that gives you different products based on which button you press. The Singleton pattern is about having only ONE vending machine shared by everyone. Combining them: one shared factory that creates different products on demand. The factory itself is the Singleton; the products it creates are not. Make this distinction clear.

What separates good from great: Two historical facts that show real depth. First: before C++11 (2011), the C++ static local variable trick was NOT thread-safe. Two threads could race to initialize the variable simultaneously ‚Äî both entering the initialization block at once. This was a silent bug that caused crashes and corruption in production codebases for years. C++11 fixed this by adding "magic statics" ‚Äî the language standard now guarantees exactly-once initialization even with concurrent access. Second: Rust is fundamentally different from C# and C++ in its approach. In C# and C++, you CAN write unsafe code ‚Äî the language will compile it and let it crash at runtime. In Rust, the compiler's type system and "borrow checker" analyze your code for data races at compile time. If you try to share mutable state between threads without proper synchronization, Rust simply REFUSES to compile your code. Safety isn't a convention you follow ‚Äî it's a compile-time guarantee. This is the most important philosophical difference between the three languages.

Q22. Your Singleton cache is returning stale data after a distributed deployment. Different pods show different values. How do you fix this? Production
Follow-up: What is cache-aside pattern and how does it relate?
How to think about it: This is the classic "distributed systems" problem with in-process caches. When your app runs on multiple servers (pods), each server has its own isolated memory ‚Äî including its own Singleton cache. Updating data on Server A doesn't automatically update Server B's cache. Think of it like 5 post offices each keeping their own address book ‚Äî if one updates an address, the others still have the old one. Describe three strategies, each with different tradeoffs between simplicity and consistency.

Root Cause: Each server (pod) has its own Singleton cache living in its own memory. When Server A updates a product and clears its own cache, servers B through E still have the old cached value. Users who hit Server B see stale data. The inconsistency only resolves when each server's cache entry expires on its own schedule ‚Äî which could be minutes or hours away.

Solution 1 ‚Äî Accept Eventual Consistency: Use short expiry times (30-60 seconds on cache entries). Accept that data might be up to 60 seconds stale. This is the simplest solution and is totally acceptable for data where "slightly old" is fine ‚Äî product descriptions, feature flags, pricing that doesn't change mid-transaction. The data eventually becomes consistent once every server's cache expires. This trade-off has a name: eventual consistency ‚Äî meaning all copies of the data will converge to the same value over time, just not necessarily at the exact same moment.

Solution 2 ‚Äî Distributed Cache (Redis): Replace the in-process IMemoryCache with a shared external cache (Redis). Now ALL servers read from and write to the SAME cache store ‚Äî like replacing 5 separate address books with one shared Google Contacts account. When you invalidate a key in Redis, every server immediately sees the change. Stronger consistency, but adds a network hop to every cache operation.

Solution 3 ‚Äî Cache Invalidation via Message Bus: When one server updates data, it broadcasts a "this key is now stale" message to all other servers via a message queue (like Azure Service Bus or RabbitMQ). Every server listens for these messages and immediately removes the affected key from its own local cache. This gives you fast local reads (no network hop to Redis) AND near-immediate consistency when data changes. More complex to set up, but best of both worlds.

What separates good from great: Knowing the three caching strategies by name. Cache-Aside (Lazy Loading) ‚Äî the most common pattern. Q24. Your circuit breaker Singleton is not tripping even after 100 consecutive failures. What's wrong? Production
Follow-up: How do you share circuit breaker state across multiple pods?
How to think about it: A circuit breaker is like an electrical circuit breaker in your home ‚Äî it trips (opens) after too many "overloads" (failures), cutting power to protect the system. For it to trip, it needs to REMEMBER each failure. If each request gets its own fresh circuit breaker with memory wiped clean, the failure count never accumulates and the breaker never trips. Walk through the logic and you'll arrive at the answer: this is the Singleton vs Transient lifetime bug. Walk through the diagnosis and fix clearly.

Root Cause ‚Äî Transient Registration (most common): The circuit breaker is registered as Transient ‚Äî meaning the DI container creates a brand new instance for every single request. Each new instance starts with a failure count of zero. Request 1 fails ‚Äî counter goes to 1, then the instance is thrown away. Request 2 creates a NEW instance with counter = 0, fails, counter = 1, thrown away again. The circuit breaker never reaches the threshold because it never remembers previous failures. The fix is obvious once you understand it: the circuit breaker MUST be a Singleton so it shares its failure count across ALL requests.

// Multi-tenant Singleton: one service, N tenant partitions
public sealed class TenantConfigService : ITenantConfigService
{
    // Keyed by tenant ID ‚Äî each tenant has its own config snapshot
    private readonly ConcurrentDictionary<string, TenantConfig> _configs = new();
    private readonly ITenantConfigProvider _provider;
    private readonly ILogger<TenantConfigService> _logger;

    public TenantConfigService(
        ITenantConfigProvider provider,
        ILogger<TenantConfigService> logger)
    {
        _provider = provider;
        _logger = logger;
    }

    public async Task<TenantConfig> GetConfigAsync(string tenantId)
    {
        // GetOrAdd ensures single load per tenant ‚Äî thread-safe
        if (_configs.TryGetValue(tenantId, out var cached))
            return cached;

        var config = await _provider.LoadAsync(tenantId);
        _configs.TryAdd(tenantId, config);

        _logger.LogInformation("Loaded config for tenant {TenantId}", tenantId);
        return config;
    }

    // Called by background service when a tenant updates their config
    public void InvalidateTenant(string tenantId)
    {
        _configs.TryRemove(tenantId, out _);
        _logger.LogInformation("Config invalidated for tenant {TenantId}", tenantId);
    }
}

Common Pitfalls

1 God Singleton ‚Äî One Class Does Everything

The God Singleton accumulates responsibilities over time. It starts as a config reader, then grows to include logging, caching, metrics, and session management. Soon it has 50 methods and every class depends on it. Changes to the God Singleton cascade to every dependent class.

Signs You Have a God Singleton
  • More than 5 distinct responsibilities in one class
  • More than 20 methods or 500 lines
  • Injected into more than 80% of your services
  • "Every time we need a new feature, we add it here"
Fix: Decompose by Responsibility
Split into focused Singletons: IConfigService, ICacheService, ILogService, IMetricsService. Each service is registered independently, tested independently, and replaced independently. Services only inject the focused interfaces they actually need.
2 Captive Dependencies ‚Äî Singleton Capturing Shorter-Lived Services

A Singleton captures a Scoped or Transient dependency in its constructor. The captured dependency lives for the Singleton's lifetime ‚Äî not its intended lifetime. DbContext captured by a Singleton holds an open connection and a change tracker for the entire app lifetime.

Classic Failure
Singleton captures DbContext ‚Üí Stale entities returned after updates ‚Üí ObjectDisposedException on connection timeout ‚Üí $23K in failed transactions
Fixes
1. Use IDbContextFactory<T> ‚Äî creates fresh contexts per operation.
2. Use IServiceScopeFactory ‚Äî creates a scope per operation and resolves Scoped services from it.
3. Change to Scoped lifetime if the service doesn't truly need to be Singleton.
4. Enable ValidateScopes = true in development to catch this at startup.
3 Untestable Static Singleton ‚Äî Hidden Global State

A hand-rolled static Singleton accessed via MySingleton.Instance creates hidden global state. Unit tests cannot substitute a test double ‚Äî the class is hardwired to use the concrete Singleton. Tests become order-dependent: state from test A bleeds into test B.

Signs
Tests pass individually but fail when run together. Tests behave differently in CI (which runs all tests) vs locally (which runs specific tests). You add internal static void ResetForTesting() hacks or use reflection to set _instance = null in test teardown.
Fix: DI + Interface
Register via AddSingleton<IService, Service>(). Inject IService into constructors. Tests inject Mock<IService>. Each test creates its own DI scope ‚Äî no shared static state between tests.
4 Unbounded Collections ‚Äî Memory Leak

A Singleton holds a ConcurrentDictionary, List, or Queue that grows without bound. Because the Singleton lives for the app's lifetime, so does the collection. After 3 months in production: 4.8GB heap, 400ms GC pauses, cascading timeouts.

Common Offenders
Cache without eviction policy. Event log without rotation. Request audit trail without maximum size. User session dictionary without expiration. All grow forever if the Singleton is never restarted.
Fix: Always Use Bounded Collections
Replace ConcurrentDictionary with IMemoryCache (SizeLimit + expiration). Replace Queue<T> with Channel<T> (BoundedChannelOptions with FullMode). Set maximum sizes and expiration on every Singleton collection.
5 Mutable Configuration ‚Äî IOptions vs IOptionsMonitor

Using IOptions<T> in a Singleton reads configuration once at startup and never picks up changes. After a config push (changing a rate limit, a feature flag, a connection timeout), the Singleton still serves the old values until the process restarts.

Distributed Singleton Problems
  • Rate limiter: 5 pods √ó 100 req/sec = 500 req/sec actual rate (not 100)
  • Circuit breaker: pod A trips circuit, pod B still sending requests to a dead service
  • Feature flags: pod A has old flags from before restart, pod B has new flags
  • Audit log: events distributed across 5 pods ‚Äî incomplete audit trail per pod
Fix: Externalize Shared State
Rate limiting: Redis INCR + EXPIRE. Circuit breaker: Redis-backed state machine. Feature flags: centralized flag service (LaunchDarkly). Audit log: centralized log aggregator (Elasticsearch). The in-process Singleton becomes a lightweight local client to the shared store.
7 Testing with Shared Static State ‚Äî Test Pollution

Static Singleton state persists between test methods in the same test run. Test A populates the cache. Test B expects an empty cache but finds Test A's data. Tests become order-dependent and fail intermittently in CI when test execution order differs from local.

// Pattern 2: For hand-rolled Singletons, expose a reset method for testing only
public sealed class ConfigSingleton
{
    private static ConfigSingleton? _instance;
    // ‚ö†Ô∏è NOT thread-safe ‚Äî acceptable only for test scenarios
    public static ConfigSingleton Instance => _instance ??= new();

    // Only for testing ‚Äî resets shared state between tests
    internal static void ResetForTesting() => _instance = null;
}
Best Fix: DI Throughout
The best fix is prevention: use DI-registered Singletons with interfaces from the start. Tests create a new IServiceCollection per test, register mock implementations, and get a fresh scope. No static state is shared between tests. No reset methods needed.
